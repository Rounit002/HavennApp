# Havenn — Complete Project Setup & Reproducibility Guide

> Goal: a developer (or AI) can clone this repository and recreate everything from
> scratch — backend, web frontend, and the signed Android APK/AAB — without asking
> any questions.
>
> This document was generated by inspecting the actual project files. Where the
> codebase contradicts older docs, the **code/config is treated as the source of
> truth** and the discrepancy is flagged.

---

## IMPORTANT: Read this first (what this project actually is)

Some prior docs and the original task template assume an **Expo / EAS / React
Native** app. **That is not what Havenn is.** The real stack is:

- A **React + TypeScript web SPA** built with **Vite** (`Frontend/`).
- A **Node.js + Express + PostgreSQL** backend (`Backend/`) that also serves the
  compiled SPA for the web.
- An **Apache Cordova** Android wrapper (`havenn/`) that bundles the same SPA and
  injects native plugins (camera, permissions, RevenueCat billing).

Consequences for this guide:

- There is **no `eas.json`, no `app.json`, no Expo, and no EAS Build.** APK/AAB are
  produced with the **Cordova + Gradle** toolchain.
- There is **no Firebase** anywhere in the project (verified — no `firebase`
  dependency, no `google-services.json`). The "Firebase Setup" section below is
  kept only to record that it is intentionally **Not Applicable**.
- In-app subscriptions use **RevenueCat** (`cordova-plugin-purchases`) as the
  primary system, with a **legacy direct Google Play Billing** path retained in the
  backend for reference.

## 1. Project Overview

### What Havenn does
Havenn is a **multi-tenant study-space / library management platform**. Each library
(tenant) gets isolated data and can manage:

- Students & memberships (active/inactive/expired/expiring), attendance via QR/barcode
- Seats, schedules/shifts, branches, lockers
- Transactions, advance payments, collections/dues, expenses, profit & loss
- Products, announcements, public student registration links, admission requests
- Hostel module (separate students, branches, collections, dashboard)
- Staff/user management with permissions
- Subscriptions/billing (premium gating) via RevenueCat on Android

There are three user types: **Owner** (tenant account), **Staff/Admin** (per-library
users), and **Student** (end users).

### Tech stack
| Layer | Technology |
|-------|-----------|
| Backend | Node.js, Express 4, PostgreSQL (`pg`), `express-session`, `winston`, `helmet`, `multer`, `cloudinary`, `googleapis`, `node-cron`, `@sendinblue/client` (Brevo) |
| Frontend | React 18, TypeScript, Vite 5, React Router 6 (HashRouter), TanStack React Query 5, Tailwind CSS 3, shadcn/ui (Radix UI), Framer Motion, Axios, Recharts, jsPDF/html2canvas, html5-qrcode |
| Mobile | Apache Cordova (`cordova-android ^15`), plugins: camera, filepath, android-permissions, **purchases (RevenueCat)** |
| Billing | RevenueCat (primary, Android), Google Play Billing direct (legacy backend) |
| Media | Cloudinary (image uploads) |
| Email | Brevo / Sendinblue |
| Hosting | Render (`https://havennapp.onrender.com`) for backend + web |

### Architecture overview
```
[React SPA (Vite, HashRouter)] --HTTP + session cookie--> [Express API] --> [PostgreSQL]
        |                                                       |        \
        |                                                       |         \-- [Cron jobs / Brevo emails]
        |                                                       |
  [Cordova Android WebView] -- same SPA bundled in www/ --> (CORS + session, RevenueCat plugin)
```
- The Express server serves the compiled SPA from `Backend/dist` for the web.
- The Cordova app bundles the **same** compiled SPA in `havenn/www/` and serves it
  from `https://localhost` so the Cordova bridge + RevenueCat plugin are injected.
- The packaged mobile app always calls the production API at
  `https://havennapp.onrender.com/api` (see `Frontend/src/utils/apiConfig.ts`).

See `ARCHITECTURE.md` for the deep technical flow (auth, tenancy, request flows).

### Platform targets
- **Web**: served by Express from `Backend/dist`.
- **Android**: Cordova APK/AAB. Package id `com.havenn.studyspace`.
  - `minSdkVersion` 23, `targetSdkVersion` 36, `compileSdkVersion` 36 (per
    `havenn/config.xml`). AndroidX enabled.
- **iOS**: not configured (no `ios/` Cordova platform present).

### Why each important folder exists
- **`Backend/routes/`** — each file is an Express router factory for one domain
  (e.g., `students.js`, `branches.js`, `revenueCatSubscriptions.js`). `server.js`
  mounts them under `/api/*` with auth/tenancy/subscription middleware. Add new API
  surface here.
- **`Backend/migrations/`** — ordered, idempotent SQL migrations. Run them in order
  to build the schema. `schema/create_all_tables.sql` is the consolidated schema.
- **`Backend/utils/`** — cross-cutting helpers: cron jobs (membership reminders,
  disabled by default), Brevo email, password hashing, WhatsApp helper.
- **`Backend/dist/`** — the compiled SPA. Created by the build; the Express static
  handler serves it with a wildcard fallback to `index.html`.
- **`Frontend/src/pages/`** — one component per route/screen.
- **`Frontend/src/components/`** + **`components/ui/`** — shared components; `ui/`
  holds shadcn/ui primitives generated via `components.json`.
- **`Frontend/src/context/`** — global React state: `AuthContext` (session
  lifecycle), `RevenueCatContext` (subscription/entitlement lifecycle).
- **`Frontend/src/services/`** — `api.ts` is the axios client (withCredentials);
  `revenueCatService.ts` wraps the Cordova `window.Purchases` global (no-op on web).
- **`Frontend/src/utils/`** — `apiConfig.ts` resolves the API base URL per platform;
  `platformUtils.ts` exposes `isCordova`/`isWeb`/`getPlatform()`.
- **`havenn/www/`** — the SPA copied from `Frontend/dist` by `npm run sync:www`.
  This is what Cordova serves locally so the native bridge is injected.
- **`havenn/platforms/`** — generated native Android project. **Git-ignored and
  wiped** by `cordova platform rm`; see `PLATFORM_ANDROID_CUSTOMIZATIONS.md`.

## 4. Project Installation

```bash
# 1. Clone
git clone <your-repo-url> HavennMain
cd HavennMain

# 2. Backend dependencies
cd Backend
npm install

# 3. Frontend dependencies
cd ../Frontend
npm install

# 4. Cordova wrapper dependencies (only needed for Android builds)
cd ../havenn
npm install
```

Create the database and load the schema:
```bash
# Create a database named to match DB_NAME in Backend/.env (default: library_management)
createdb library_management            # or use psql / pgAdmin

# Load the consolidated schema
psql -d library_management -f Backend/schema/create_all_tables.sql

# Then apply migrations in numeric order (001 -> 011). Example:
psql -d library_management -f Backend/migrations/001_multi_tenant_setup.sql
# ...continue through...
psql -d library_management -f Backend/migrations/011_add_revenuecat_fields.sql
```
> Some `.js` files in `Backend/` and `Backend/migrations/` are one-off runners
> (e.g., `run_migration.js`, `run_gender_migration.js`,
> `migrations/run_admission_requests_migration.js`). They read `Backend/.env` and
> run a specific migration via `node`. Prefer the raw `.sql` files for a clean,
> reproducible setup.

Configure environment files (see Section 5), then run:
```bash
# Terminal 1 — backend API on :3000
cd Backend
npm run dev            # nodemon server.js   (or: npm start  -> node server.js)

# Terminal 2 — frontend dev server on :8080 (proxies /api -> :3000)
cd Frontend
npm run dev
```
Open http://localhost:8080. The Vite dev proxy forwards `/api` to the backend so
session cookies stay same-origin.

Build the web bundle and have the backend serve it (production-style):
```bash
cd Backend
npm run build          # installs deps, builds Frontend, copies Frontend/dist -> Backend/dist
npm start              # serves API + SPA from one process
```

`firebase` / `@react-native-firebase` dependency, no `google-services.json`, no
Firestore/Storage/FCM usage anywhere.

If Firebase is added later, this section should document project creation, Android
app registration, SHA-1/SHA-256 keys, `google-services.json` placement under the
Android platform, and Auth/Firestore/Storage enablement. None of that exists today.

## 7. RevenueCat Setup (PRIMARY subscription system)

RevenueCat is wired through `cordova-plugin-purchases@^8.0.7` on Android. On web the
SDK is never loaded and premium is derived from the backend `is_subscription_active`
flag.

### 7.1 Dashboard setup
1. Create an account at https://www.revenuecat.com and create a **Project** (e.g., "Havenn").
2. Add an **App** of type **Google Play / Android**, package name `com.havenn.studyspace`.
3. Upload the **Google Play service account JSON** to RevenueCat (Project → App →
   Google Play credentials) so RevenueCat can validate purchases server-side.

### 7.2 Products
- In RevenueCat → **Products**, import/create the product(s) that map to your Play
  Console subscription (e.g., product id `havenn_premium_monthly`).

### 7.3 Entitlements
- In RevenueCat → **Entitlements**, create an entitlement and note its **Identifier**
  (the short slug, not the display name).
- This identifier MUST match both `VITE_REVENUECAT_ENTITLEMENT_ID` (frontend) and
  `REVENUECAT_ENTITLEMENT_ID` (backend). Default convention in code is `premium`.
- Attach the product(s) to this entitlement.

### 7.4 Offerings
- In RevenueCat → **Offerings**, create an offering (the "current" one is used by
  default) and add a package containing your product. To force a specific offering,
  set `VITE_REVENUECAT_OFFERING_ID`.

### 7.5 API keys
- **Public Android SDK key** (starts with `goog_`, or `test_` for the Test Store) →
  `Frontend/.env` `VITE_REVENUECAT_ANDROID_API_KEY`.
- **Secret API key** (v1, `sk_...`, keep private) → `Backend/.env`
  `REVENUECAT_SECRET_API_KEY`. Used for server-to-server subscriber lookups.

### 7.6 Webhook
- RevenueCat → **Integrations → Webhooks**: set the URL to
  `https://<your-backend>/api/subscriptions/revenuecat/webhook` and set an
  **Authorization** header value equal to `REVENUECAT_WEBHOOK_AUTH_HEADER`.
- The backend always returns 200 to avoid retry storms and reconciles renewals,
  cancellations, and expirations onto the `libraries` table.

### 7.7 Android configuration / App User ID
- The frontend identifies the user to RevenueCat using the convention
  **`havenn_<libraryId>`**, so all events/subscriptions map back to the correct
  tenant. This is handled in `Frontend/src/context/RevenueCatContext.tsx`.
- The `com.android.vending.BILLING` permission is declared in `havenn/config.xml`.

### 7.8 Testing subscriptions
1. Build and upload a signed build to a Play Console testing track (Section 13).
2. Add tester Gmail accounts (Play Console → testers) — sandbox purchases require it.
3. Install the app from the testing track on a device signed in with a tester account.
4. Open the paywall (Navbar "Upgrade" for owner/admin), purchase, and confirm:
   - In Chrome `chrome://inspect` console on the device:
     `typeof window.Purchases` → `"object"` (RevenueCat bridge present).
   - After purchase, the app calls `POST /api/subscriptions/revenuecat/sync` which
     updates the DB immediately (no webhook wait).
5. For pure SDK testing without real money, use the **Test Store** key (`test_...`).

### Relevant code paths
- Service: `Frontend/src/services/revenueCatService.ts`
- Context/lifecycle: `Frontend/src/context/RevenueCatContext.tsx`
- Paywall UI: `Frontend/src/components/Paywall.tsx`, `PaywallContent.tsx`
- Backend: `Backend/routes/revenueCatSubscriptions.js`
  - `POST /api/subscriptions/revenuecat/sync` (owner session)
  - `POST /api/subscriptions/revenuecat/webhook` (server-to-server)
- DB fields: migration `Backend/migrations/011_add_revenuecat_fields.sql`
  (`subscription_expires_at`, `revenuecat_app_user_id`).

## 8. Google Play Billing Setup

Google Play is required as the **store backing RevenueCat**, and the backend also
keeps a **legacy direct Billing integration** (`Backend/routes/googlePlaySubscriptions.js`)
for reference/fallback.

### 8.1 Create the subscription in Play Console
1. Play Console → your app (`com.havenn.studyspace`) → **Monetize → Products →
   Subscriptions**.
2. Create a subscription with **Product ID** `havenn_premium_monthly` (matches
   `GOOGLE_PLAY_PRODUCT_ID` / `VITE_GOOGLE_PLAY_PRODUCT_ID`).
3. Add a **Base plan** (e.g., monthly, auto-renewing) and set price/region.
4. Optionally add **Offers** (intro/free-trial). Activate the base plan.

### 8.2 Product IDs / base plans / offers
- Product ID: `havenn_premium_monthly` (keep identical across Play, RevenueCat, and env vars).
- Base plan id and offer ids are configured in Play Console and surfaced through the
  RevenueCat offering/package.

### 8.3 Service account (for backend verification & RevenueCat)
1. Google Cloud Console → enable **Google Play Android Developer API**.
2. Create a **service account**, download its **JSON key**.
3. Play Console → **Setup → API access** → link the service account and grant
   permission to view financial data / manage orders.
4. Provide the JSON to the backend via `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON` (inline,
   escaped newlines) or `GOOGLE_PLAY_SERVICE_ACCOUNT_PATH`, and upload the same JSON
   to RevenueCat.

### 8.4 Testing accounts
- Play Console → **Testing** track → add tester Gmail accounts (also add them to a
  License testing list under Setup → License testing for sandbox pricing).

### 8.5 Legacy billing integration endpoints
- `POST /api/subscriptions/google-play/verify` (owner session) — verifies a purchase token.
- `POST /api/subscriptions/google-play/webhook` — optional Pub/Sub push handler.
- `GET  /api/subscriptions/google-play/health` — service-account sanity check.
- DB fields: migration `Backend/migrations/010_add_google_play_fields.sql`.

> For new work, prefer the RevenueCat path (Section 7). The direct Google Play path
> is retained but superseded.

## 9. APK Build Process

APKs are produced with **Cordova + Gradle** (not EAS). All commands run from the
`havenn/` directory. Ensure `JAVA_HOME` points to JDK 17 in a fresh terminal.

### One-time / after a clean checkout
```bash
cd havenn
npm install
cordova platform add android      # cordova-android ^15 per package.json
# (If a build later fails, see havenn/PLATFORM_ANDROID_CUSTOMIZATIONS.md)
```

### Build helpers (defined in havenn/package.json)
```jsonc
"build:web":            "cd ../Frontend && npm run build",          // Vite build -> Frontend/dist
"sync:www":             "rimraf www && cpr ../Frontend/dist www",   // copy SPA into havenn/www
"build:android":        "npm run build:web && npm run sync:www && cordova build android",
"build:android:release":"npm run build:web && npm run sync:www && cordova build android --release"
```

### Development / debug APK (unsigned debug)
```bash
cd havenn
npm run build:android
# Output: havenn/platforms/android/app/build/outputs/apk/debug/app-debug.apk
```

### Preview / internal-test APK (signed release APK)
A signed release APK uses the keystore in `havenn/build.json`:
```bash
cd havenn
npm run build:android:release
# Output: havenn/platforms/android/app/build/outputs/apk/release/app-release.apk
```

### Production APK
For production, Google Play requires an **AAB** (Section 10), not an APK. A signed
release APK (above) is suitable for direct distribution / sideloading and for
internal QA on devices.

### Verify the build on a device
Install and remote-debug via `chrome://inspect`, then in the device console:
```js
typeof window.cordova   // "object"
typeof window.Purchases // "object"  (RevenueCat bridge loaded)
```
Both `"object"` means the SPA loaded from the local Cordova shell and billing works.

## 10. AAB (Android App Bundle) Build Process

Google Play uploads use an **AAB**. Build it with Cordova/Gradle in release mode.

```bash
cd havenn
npm run build:web        # build SPA
npm run sync:www         # copy into www/
cordova build android --release -- --packageType=bundle
# Output: havenn/platforms/android/app/build/outputs/bundle/release/app-release.aab
```

### Signing
- Signing is driven by `havenn/build.json`:
  ```json
  {
    "android": {
      "release": {
        "keystore": "my-release-key.keystore",
        "storePassword": "********",
        "alias": "myalias",
        "password": "********",
        "keystoreType": ""
      }
    }
  }
  ```
- The keystore file `havenn/my-release-key.keystore` (alias `myalias`) signs both the
  release APK and the AAB. **Keep this keystore and its passwords safe and out of
  git** — losing it means you cannot update the app on Play.
- Recommended: enroll in **Google Play App Signing** so Google manages the final app
  signing key while you keep an upload key.

### Upload requirements
- The AAB must use a **versionCode higher than any previously uploaded build**
  (`android-versionCode` in `config.xml`, currently `10003`).
- It must be signed with the registered upload/app-signing key.
- `targetSdkVersion` must meet Play's current minimum (36 here, comfortably above).

## 13. Deployment Process

### Backend + Web (Render)
The web app and API are served by the same Express process (`Backend/server.js`),
deployed to Render at `https://havennapp.onrender.com`.

1. Provision PostgreSQL (Render/Railway/managed PG). Set `DATABASE_URL` with SSL.
2. Set all `Backend/.env` variables in the host's environment settings.
3. Build command: `npm run build` (from `Backend/`) — installs deps, builds the
   Frontend, and copies `Frontend/dist` → `Backend/dist`.
4. Start command: `npm start` (`node server.js`).
5. Ensure session cookies are `secure` + `SameSite=None` behind HTTPS (handled in
   `server.js` for production) so WebViews keep the session.

### Android release workflow (Play Console)
1. **Bump the version**: edit `havenn/config.xml`:
   - increase `android-versionCode` (must be greater than the last uploaded value,
     currently `10003`),
   - update `version` (version name) as appropriate.
2. **Build the production AAB** (Section 10):
   ```bash
   cd havenn
   npm run build:web && npm run sync:www
   cordova build android --release -- --packageType=bundle
   ```
3. **Upload** `app-release.aab` to Play Console.
4. **Internal testing**: create/refresh an Internal testing release, add tester
   emails, share the opt-in link, install and smoke-test (esp. subscriptions).
5. **Closed testing**: promote to a Closed track for a wider tester group; complete
   any required pre-launch checks.
6. **Production release**: promote to Production, set rollout percentage, and submit
   for review.

> Keep the keystore/upload key consistent across releases. If Play App Signing is
> enabled, you upload with the upload key and Google re-signs with the app key.

## 14. Common Issues and Fixes

### Android build failures
- **Gradle picks JDK 11 → build fails (AGP requires 17).** Set `JAVA_HOME` to JDK 17
  and open a fresh terminal (`java -version` → 17). Fallback: add
  `org.gradle.java.home=C:/Program Files/Eclipse Adoptium/jdk-17.0.16.8-hotspot` to
  `platforms/android/gradle.properties`. See `PLATFORM_ANDROID_CUSTOMIZATIONS.md`.
- **`CordovaLib` fails to compile referencing API-36 symbols.** Install **SDK
  Platform 36 + Build-Tools 36.0.0**; `compileSdkVersion` must be 36 (per `config.xml`).
- **`platforms/android` edits disappear.** That folder is git-ignored and wiped by
  `cordova platform rm`. Re-apply changes via `PLATFORM_ANDROID_CUSTOMIZATIONS.md`.
- **Duplicate Google Play Billing class / duplicate-class build error.** Only ONE
  billing plugin may exist. `cordova-plugin-purchase` (legacy) was removed; keep only
  `cordova-plugin-purchases` (RevenueCat). Don't reinstall the old one.
- **Windows reserved file `platforms/android/CON` breaks copy/delete tooling.**
  Delete it: `del "\\?\D:\WebSoftware\HavennMain\havenn\platforms\android\CON"`.
- **Gradle wrapper version mismatch.** If a fresh `platform add` pulls Gradle 8.7 and
  fails, pin `distributionUrl` to `gradle-8.4-all.zip` in
  `platforms/android/gradle/wrapper/gradle-wrapper.properties`.

### Firebase issues
- N/A — Firebase is not used. If you hit Firebase-related instructions elsewhere,
  they do not apply to this project.

### RevenueCat issues
- **`window.Purchases` is `undefined` on device.** The SPA isn't loading from the
  local Cordova shell. Confirm `config.xml` has `<content src="index.html" />` and
  that `www/` was populated by `npm run sync:www`.
- **Offerings empty / paywall shows nothing.** Verify the offering has a package and
  is set "current", products are attached to the entitlement, and
  `VITE_REVENUECAT_ANDROID_API_KEY` is correct (prod `goog_` vs sandbox `test_`).
- **Entitlement never activates after purchase.** Ensure
  `VITE_REVENUECAT_ENTITLEMENT_ID` (frontend) === `REVENUECAT_ENTITLEMENT_ID`
  (backend) === the RevenueCat entitlement **Identifier**. (The two `.env.example`
  files currently disagree — fix this.)
- **Webhook not updating DB.** The `Authorization` header in RevenueCat must equal
  `REVENUECAT_WEBHOOK_AUTH_HEADER`; the endpoint is
  `/api/subscriptions/revenuecat/webhook`.

### Play Billing issues
- **Purchases fail in testing.** Tester Gmail must be on the testing track AND on the
  License testing list; the app must be installed from Play (not sideloaded) for real
  sandbox purchases.
- **Verification 401/permission errors.** Re-check the service-account JSON, that the
  Play Android Developer API is enabled, and the account is linked in Play Console →
  API access with sufficient permissions.

### Android SDK issues
- **`ANDROID_HOME`/SDK not found.** Set `ANDROID_HOME` to the SDK path and add
  `platform-tools` to `PATH`. Install missing platforms/build-tools via SDK Manager.
- **`adb` not recognized.** Add `platform-tools` to `PATH`.

### Gradle issues
- **OOM / heap dumps (`hs_err_pid*.log`).** `gradle.properties` already sets
  `-Xmx4g`. Close other apps; delete stale crash logs in `platforms/android/`.

### EAS issues
- N/A — this project does not use EAS/Expo. Ignore any EAS instructions.

### Backend / web issues
- **DB connection fails.** Verify `DB_*` (or `DATABASE_URL` with SSL in prod) and that
  the schema + migrations were applied. Startup logs masked DB details via winston.
- **Session/cookie lost in WebView.** Ensure HTTPS and `secure`+`SameSite=None`
  cookies in production; CORS must allow credentials and the `file://`/`null`/local
  origins (already configured in `server.js`).
- **SPA 404s on refresh.** The wildcard fallback to `dist/index.html` requires the
  SPA to be built into `Backend/dist` (`npm run build`).
